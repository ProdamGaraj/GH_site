### Цель
Создать современный, высокопроизводительный и удобный сайт строительной компании с:
- быстрым администрированием через CMS;
- автоматическим обновлением данных по квартирам/планировкам/остаткам из CRM;
- мультиязычностью (RU/UZ);
- SEO (ЧПУ, мета, микроразметка);
- интеграциями (аналитика, IP-телефония, чат-бот);
- высокой скоростью (ориентир: загрузка ключевых страниц до ~2,5 сек).

### Формат работы
- Итерации: спринты по 2 недели.
- Общая длительность: 10–12 недель (5–6 спринтов) + 3–5 дней на подготовку.
- Среда: dev/staging/prod.

### Условности и допущения (чтобы план был реалистичным)
- CRM предоставляет API/экспорт по объектам, квартирам, ценам, статусам “в продаже”, планировкам/изображениям.
- Есть ответственный со стороны CRM/интеграций (доступы, лимиты, тестовая среда).
- Есть брендбук/гайд, исходники дизайн-макетов или возможность оперативного согласования.
- Контент (тексты, фото/рендеры, планы, юридические тексты) поставляется параллельно разработке.

## Структура сайта (как ориентир для бэклога)
Основные разделы:
- Главная
- О компании
- Жилые комплексы (каталог) + карточка ЖК
- Коммерческие объекты (каталог) + карточка объекта
- Новости и акции
- Карьера и вакансии
- Партнеры
- Публичная оферта
- Контакты (карта)
- Социальные сети (чат-бот)

Ключевая страница: карточка проекта (ЖК/объект) с блоками:
- Галереи: главное фото (карусель), общие фото, фото МОПов
- Расположение на карте
- Особенности (стандартизированные значки)
- Типы оплаты
- Характеристики (класс, этажность, блоки, диапазон квадратур, доступные квартиры — авто, скидка до)
- Квартиры (планировка, площадь, комнатность, цена) — авто из CRM
- Фильтрация (комнаты, цена, этаж)
- Инфраструктура (карточки с фото и названием)

## Definition of Done (общий, чтобы контролировать качество)
Считаем фичу готовой, если:
- есть RU и UZ (если применимо) и корректные языковые URL;
- адаптивность (мобильный/планшет/десктоп) проверена;
- учтены SEO-атрибуты (title/description/OG, каноникал при необходимости);
- формы отправляют лид в CRM, передают UTM и фиксируют источник;
- есть базовая защита (валидация, rate limit/captcha по необходимости);
- критические сценарии покрыты тест-кейсами и пройдены на staging.

## Спринт 0 (3–5 дней): Подготовка и фиксация требований

### Цели
- Зафиксировать структуру сайта, типы страниц, блоки и контент-модель.
- Прояснить интеграции (CRM/телефония/аналитика/чат) и “контракт данных”.
- Утвердить нефункциональные требования (скорость, безопасность, SEO, доступность).

### Ценность (business value)
- Резко снижает риск переделок и сдвигов сроков на интеграциях и контенте.

### Ключевые задачи
- Воркшоп: карта страниц + навигация + сценарии (просмотр проектов, подбор квартиры, отправка заявки).
- UX-прототипы ключевых шаблонов: главная, каталог, карточка проекта, контакты, формы.
- Спецификация CRM:
	- какие сущности и поля нужны (объект, корпус/секция, квартира, планировка, цены, статусы);
	- как фильтровать “только в продаже”;
	- частота обновления и ограничения (лимиты, SLA, webhooks/экспорт).
- Спецификация лидов в CRM:
	- обязательные поля (телефон, имя, выбранный проект/квартира);
	- UTM-метки (utm_source/medium/campaign/content/term) + дополнительные (gclid/yclid при необходимости).
- SEO-скелет:
	- правила ЧПУ для каталогов/карточек;
	- sitemap, robots, hreflang;
	- шаблоны мета-тегов.

### Артефакты/результаты
- Утверждённая структура сайта и список шаблонов страниц.
- Контент-матрица (страница → блоки → поля CMS → кто заполняет).
- API-спека по CRM (минимум: описание эндпойнтов/форматов/примеров).
- Backlog MVP с приоритетами и критериями приемки.

## Спринт 2 (2 недели): Каталоги проектов + CRM-интеграция по квартирам (v1)

### Цели
- Реализовать каталоги “Жилые комплексы” и “Коммерческие объекты”.
- Реализовать карточку проекта с обязательными блоками.
- Подключить автоматическое получение данных по квартирам “в продаже” из CRM.

### Ценность
- Ключевой sales-функционал: актуальные квартиры/цены/остатки без ручных обновлений.

### Ключевые задачи
- Карточка проекта (обязательные блоки):
	- галереи (главная карусель, общие фото, МOПы);
	- карта расположения;
	- особенности (иконки/значки);
	- типы оплаты;
	- характеристики проекта (включая “количество доступных квартир” и “скидка до”);
	- инфраструктура (фото + название).
- Блок “Квартиры”:
	- карточки квартиры (планировка/площадь/комнатность/цена);
	- фильтры по комнатности/цене/этажу;
	- пагинация/виртуализация списка при больших объёмах.
- CRM-адаптер:
	- нормализация данных (CRM → внутренний формат);
	- правило “только в продаже”;
	- кеширование и TTL;
	- обработка ошибок (что показываем пользователю при недоступной CRM).
- Лиды в CRM:
	- формы заявок отправляют данные в CRM;
	- фиксируются UTM + контекст (страница/проект/квартира).

### Результаты
- Каталоги + карточки проектов работают на staging.
- Данные по квартирам автоматически приходят из CRM и отображаются.
- Фильтры работают корректно и быстро.
- Формы уходят в CRM (или в тестовую CRM-среду) вместе с UTM.

## Рекомендуемый стек и подходы

### Вариант A (рекомендованный): Headless CMS + Next.js (максимум скорости/SEO/контроль интеграций)
- Frontend: Next.js (SSR/SSG/ISR), TypeScript.
- CMS: Strapi / Sanity / Contentful (выбор по бюджету и потребности в workflow).
- Интеграции CRM: отдельный слой (API routes/микросервис/serverless) + кеш (Redis/Upstash).
- CDN/WAF: Cloudflare (или аналог), кеширование и защита.
- Наблюдаемость: Sentry (ошибки), APM/логи по необходимости.

Почему подходит под требования:
- Скорость: ISR/edge + CDN + оптимизация изображений.
- SEO: SSR и полный контроль мета/URL.
- CRM: отдельный адаптер снижает связность и упрощает ретраи/кеш.
- RU/UZ: зрелые библиотеки i18n + локализуемый контент в CMS.

### Вариант B: Классическая CMS (быстрее старт с “встроенной админкой”, но выше риск по 2,5 сек)
- CMS: 1C‑Bitrix (часто выбирают под интеграции/роли) или WordPress/Drupal.
- Интеграции: модули/плагины + cron/очереди, CDN и кеширование.

Когда уместен:
- если команда/подрядчик сильнее в конкретной CMS;
- если требуется максимально типовая админка “из коробки”.
